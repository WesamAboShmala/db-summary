

# 🌐 محاضرة Flashback Database (Part 2)

المحاضرة اللي فاتت خلصنا فيها: الفلاش باك عالمستوى التيبل، وعالمستوى الترانزكشن، وعالمستوى الفيرجنز (كيف بشوف التغيرات على مستوى الداتا). هلأ هاي المحاضرة موضوعها: **الفلاش باك على مستوى الداتابيز ككل**.

### 💭 تخيلي هذا السيناريو:

عندك داتابيز، وفيها يوزر (مستخدم) هو أساسي جداً، وكل البرنامج شغال عليه. غلط، مبرمج أو DBA عمل:

```sql
DROP USER main_user;

```

يعني مسح هاد اليوزر بالكامل. النتيجة: البرنامج كله وقف عن الشغل، لأنه ما في يوزر يشبك (Connect) فيه.

هاي بالضبط **Logical Error** يعني: الأمر اللي كتبته (`DROP USER`) مكتوب صح، ما في خطأ إملائي أو نحوي. بس القرار البرمجي نفسه كان غلط يعني حذفت شي ما كان المفروض تحذفه.

> 📁 *(وفيه نوع تاني اسمه **Physical Error** تلف فعلي بالهاردسك أو بالملفات، وهاد موضوع تاني كلياً، ما بينحل بالفلاش باك، بينحل بموضوع لاحق اسمه **Recovery**).*

---

## ⚖️ 3. ليش نستخدم الفلاش باك بدل الحل التقليدي؟

الحل التقليدي كان: خدلي نسخة احتياطية قديمة (Backup) وارجع منها. بس هاد الحل عنده مشكلتين كبار:

* ⏳ **بياخذ وقت طويل:** نص ساعة على الأقل.
* 📉 **بيخسّرك بيانات كتير:** لأنه لو النسخة كانت من امبارح، كل شغل اليوم بضيع، مش بس الغلطة اللي صارت.

#### 🚀 الفلاش باك أحسن بكتير لأنه:

* ⏱️ بياخذ وقت أقل بكثير.
* 🎯 بيرجع بس الفترة اللي أنا بحددها (مثلاً آخر 4 دقايق بس)، مش الداتابيز كاملة لأسبوع كامل للوراء.

⚠️ **بس فيه شرط:** لازم يكون الفلاش باك مفعّل من قبل على الداتابيز، وإلا ما فيك تستخدميه أصلاً، وبترجعي بالغصب للحل البطيء (Backup).

---

## 🛠️ 4. الخطوات: كيف أفعّل الفلاش باك مسبقاً (قبل أي غلط)

الفلاش باك على مستوى الداتابيز بيقول لي: لازم تكون مفعّل حاجة اسمها **Flashback Logs**. الخطوات بالترتيب:

### 1️⃣ الخطوة 1: طفى الداتابيز

```sql
SHUTDOWN IMMEDIATE;

```

📊 **شرح خيارات الـSHUTDOWN (معلومة جانبية مهمة أحفظها):**

| الأمر | شو بيعمل؟ |
| --- | --- |
| **`SHUTDOWN NORMAL`** | بيستنى كل السيشن تسكر بشكل طبيعي، وبعدين بيسكر. |
| **`SHUTDOWN IMMEDIATE`** | بيستنى الترانزكشن تخلص (Commit أو Rollback)، وإذا أخذت وقت أكتر، هو بروح يعمل لها Rollback بنفسه. |
| **`SHUTDOWN ABORT`** | بيسكر فوراً، مش مستعجل ينتظر — أسرع طريقة. |

### 2️⃣ الخطوة 2: شغّل الداتابيز بحالة NOMOUNT

```sql
STARTUP NOMOUNT;

```

* **شرحها:** أنا هون بشغّل الإنستنس فقط، بس الإنستنس. واللي بيلزمني عشان تشتغل الداتابيز بالـNOMOUNT State هو الباراميتر فايل فقط (`SPFILE` أو `PFILE`). يعني شغّلت الإنستنس من خلال الباراميتر فايل بدون أي شي تاني.
* 📑 *ملاحظة جانبية:* `SPFILE` = ملف Static، `PFILE` = مش Static.

### 3️⃣ الخطوة 3: حوّلها لحالة MOUNT

```sql
ALTER DATABASE MOUNT;
-- أو: STARTUP MOUNT

```

* **شرحها:** MOUNT معناته: أنا رحت شغّلت الإنستنس من خلال الباراميتر فايل، وقريت الـControl File. فصار عندي بهاي الحالة Status تبع الـControl File هو Open. لكن الـData File والـRedo Log File والـArchive Log هذول مش فعالين لسا. اللي اشتغل عندي بس الباراميتر فايل والـControl File.

### 4️⃣ الخطوة 4: فعّل الـArchive Log Mode

```sql
ALTER DATABASE ARCHIVELOG;

```

* **شو معنى الأمر هاد؟** إنه الداتابيز تبعتي صارت Archive Log. يعني لما يتم الـRedo Log Files (يعني يمتلي)، خزّن لي إياه بالـArchive.
* ❓ *ليش هاي الخطوة ضرورية؟* لازم كل الأمثلة والسامبلز اللي بدي اشتغل فيها تكون الداتابيز بالـArchive، عشان:
1. بقدر آخذ Backup وهي شغالة.
2. بقدر أعمل Restore/Recovery وهي شغالة.
3. بقدر أفعّل الفلاش باك.



🔴 **قاعدة أساسية لازم أحفظها:** عشان تشتغل الفلاش باك، لازم لي شغلتين مع بعض:

1. الفلاش باك لوجز تكون شغّالة.
2. الأركايف لوج مود يكون شغال.

> الحالتين لازم أصحيهم مع بعض.

💡 *بقدر أرجعها بأمر `ALTER DATABASE NOARCHIVELOG` بتبطل الأركايف لوج مود. بس عندها: الـRecovery Manager (RMAN) ما رح يشتغل إلا بالأركايف، والفلاش باك برضه ما رح يشتغل.*

💡 **نصيحة شخصية أحفظها:** الأصح والأحسن إنك لما تشغّل الداتابيز، تكون فاهم أي Mode أنت شغّلها فيه بيسهّل عليك بموضوع الـDBA، وبيزيد التوفّر (Availability).

### 5️⃣ الخطوة 5: فعّل الفلاش باك

```sql
ALTER DATABASE FLASHBACK ON;

```

* ⚠️ **ملاحظة من التجربة الفعلية:** قبل ما يعمل الـArchive Log، جرب هاد الأمر وما ضبط معه. ليش؟ لأنه الفلاش باك بحاجة الأركايف يكون مفعّل الأول، وبعدين الفلاش باك... الترتيب مهم!
* **شرح الأمر:** لما أفعّل الفلاش باك، بتصير الداتابيز الآن قابلة إنك ترجعها زي ما بدك.

### 6️⃣ الخطوة 6: افتح الداتابيز

```sql
ALTER DATABASE OPEN;

```

👉 **هلأ خلصت الكونفيجريشن:** حطيت الداتابيز بالـNOMOUNT State، بعدين حوّلتها للـMOUNT State، بعدين شغّلت الأركايف لوج والفلاش باك، وبعدين فتحتها... اشتغلت الداتابيز بالكونفيجريشن الجديدة الصحيحة.

## 🔄 5. الاسترجاع الفعلي لما يصير الغلط

```sql
-- تأكد من عدد اليوزرز قبل الغلط
SELECT COUNT(*) FROM dba_users;
 
-- اعمل الغلط (محاكاة Logical Error)
DROP USER test CASCADE;
 
-- تأكد إنه العدد قل (وهاي اللحظة بتنحفظ كـSavepoint)
SELECT COUNT(*) FROM dba_users;
 
-- ارجع بالفلاش باك
SHUTDOWN IMMEDIATE;
STARTUP MOUNT;
 
FLASHBACK DATABASE TO TIMESTAMP (SYSDATE - INTERVAL '4' MINUTE);
 
-- افتح بـRESETLOGS
ALTER DATABASE OPEN RESETLOGS;
 
-- تأكد إنه رجع
SELECT COUNT(*) FROM dba_users;

```

> ⚠️ **ملاحظة:** خسرت كل التغييرات اللي صارت بعد الـSavepoint.

## 🧠 6. ليش لازم RESETLOGS بالضبط؟ (الشرح التقني)

لما بتشتغل الداتابيز، عندي ثلاث فايلات لهم SCN واحد — هاد الـCheckpoint بيشتغل على الثلاثة فايلات هدول:

1. Online Redo Log
2. Control File
3. Database File

كل ما يشتغل الـCheckpoint (وهو بيشتغل كل 30 ثانية، أو عند كل عملية Commit)، بيدي **Synchronization Number (SCN)** للثلاث فايلات هدول... هاد زي هاد زي هاد، بياخذوا نفس الـSCN تبعه.

الآن أنا شو اللي عملته؟ عملت فلاش باك، رجّعت البيانات (أو رجّعت الداتابيز نسخة لورا) شي طبيعي، اختلف الـSynchronization (SCN) هون بين الثلاث ملفات.

فبعد أي عملية Recovery، عشان أشغّل الداتابيز بطريقة صحيحة، بروح بقول له Reset للـLogs هدول يعني أعطيهم رقم جديد، لأني عملت عملية Recovery. إذا تمت عملية الـRecovery بشكل صحيح، سواء فيزيكال أو لوجيكالي، فبروح الـCheckpoint هون بيعطي رقم جديد لهدول الثلاثة، فبيسمح للداتابيز إنهم يشتغلوا... فهذا هو المقصود بعملية الـ**RESETLOGS**.

---

## ⚠️ 7. تحذير مهم: كل التغييرات بترجع، مش بس غلطتي (كلام المهندس)

التغيرات اللي صارت ما بعد حذف اليوزر، على مستوى الداتابيز كلها، كلها لازم تتغير (ترجع للوراء)! فلازم لازم هون تكون فاهم، وتقول للشباب اللي بيشتغلوا (سواء بالبرامج المالية/التحصيل، أو بالبرامج الإدارية، أو أي أمور فيها بيانات Sensitive) إنه: **"في أربع دقايق، ارجعوا أعيدوا شغلكم من جديد!"**

📌 **النصيحة العملية:** كل ما قلّلت الوقت (يعني اكتشفت وعالجت الغلطة بسرعة)، كل ما قلّلت على زملائك الجهد. لو أنا بدي أرجع لنسخة باك أب من امبارح، معناته بدي أعيد كل شغلي اليوم فبخسر شغلي اليوم على مستوى المؤسسة ككل، وهاد كلام مش مقبول/مش المفروض.

## 📘 8. معلومات نظرية إضافية

* **أ) اسم تاني للموضوع:** فلاش باك داتابيز = **Point-in-Time Recovery** (استرجاع لنقطة زمنية محددة).
* **ب) متى بستخدمه بالضبط؟** بتستخدم دايماً لـ **Accidental Things** (أشياء صارت بالغلط) أو **Logical Corruption**. 🚩 هاد في اللوجيكال فقط، مش في الفيزيكال.
* **ج) وين بتتخزن الفلاش باك لوجز؟** الفلاش باك داتابيز بتعتمد دايماً على الفلاش باك لوجز، وهاي بتتخزن في **Fast Recovery Area (FRA)**. هاد المكان هو مكان الشغل بكل موضوع الـAvailability. فيه:
* Archive Logs
* Flashback Logs
* Backup Sets
* نسخة من الـControl File
* *(الـOnline Redo Log File مش موجود هون)*



#### ⚙️ د) كيف أحدد مسار وحجم الـFast Recovery Area؟

فيه Parameter اسمه:

* `DB_RECOVERY_FILE_DEST` ← المسار.
* `DB_RECOVERY_FILE_DEST_SIZE` ← الحجم.

الحجم بالديفولت حوالي 4 جيجا، بقدر أزوّده لـ 90 أو 200 جيجا حسب حجم التغييرات المتوقعة عندي. أنا اللي لازم أحدد: وين بدي اللوجز، وين بدي الأركايف لوجز، وقديش الحجم المناسب.

> ❓ **ليش نخلي كل شي بنفس المكان؟** لأنه لاحقاً (بموضوع RMAN) رح أحتاج آخذ Backup من الفلاش باك أو من الداتابيز، فلازم تكون الداتابيز بالـArchive Mode أصلاً من البداية.

## 🎯 خلاصتي الشاملة النهائية

### 💡 الفكرة الكبيرة:

* **نوعين أخطاء بس:** Logical (بينحل بالفلاش باك) و Physical (بينحل بـRecovery من Backup).
* الفلاش باك أسرع وأقل خسارة من الـRecovery العادي (دقايق مقابل نص ساعة على الأقل، ورجوع لفترة محددة مقابل رجوع لنسخة قديمة كاملة).

### 🔑 الشرط الأساسي (لازم يكون جاهز قبل أي غلط):

`Archive Log Mode مفعّل`  +  `Flashback مفعّل`

### 💻 كود التجهيز (مرة واحدة، قبل أي مشكلة):

```sql
SHUTDOWN IMMEDIATE;
STARTUP MOUNT;
ALTER DATABASE ARCHIVELOG;
ALTER DATABASE FLASHBACK ON;
ALTER DATABASE OPEN;

```

### 💻 كود الاسترجاع (بعد ما يصير الغلط):

```sql
SELECT COUNT(*) FROM dba_users;
 
DROP USER test CASCADE;
 
SELECT COUNT(*) FROM dba_users;   -- (save the time savepoint)
 
SHUTDOWN IMMEDIATE;
STARTUP MOUNT;
 
FLASHBACK DATABASE TO TIMESTAMP (SYSDATE - INTERVAL '4' MINUTE);
 
ALTER DATABASE OPEN RESETLOGS;
 
SELECT COUNT(*) FROM dba_users;   -- خسرت كل التغييرات بعد الـSavepoint

```

### ❓ ليش RESETLOGS ضروري:

لأنه الـCheckpoint بيدي رقم موحّد (SCN) لثلاث ملفات (Redo Log، Control File، Data File) كل 30 ثانية أو كل Commit. الفلاش باك بيكسر هاد التزامن، فلازم RESETLOGS يعطيهم رقم جديد موحّد عشان الداتابيز تقدر تفتح وتشتغل.

### 🛑 التحذير الأهم:

الفلاش باك عالداتابيز بيرجع كل شي خلال الفترة المحددة، مش بس غلطتي — فلازم أنبّه كل المستخدمين.

### وين بتتخزن الفلاش باك لوجز:

بالـ **Fast Recovery Area (FRA)**، مع الأركايف لوجز والباك أب سيت ونسخة الـControl File.
