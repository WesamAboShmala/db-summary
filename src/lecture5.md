## 1. التكنولوجيا اللي بيعتمد عليها كل نوع فلاش باك

لما بحكي على مستوى الداتابيز كاملة، بيعتمد على حاجة اسمها **Flashback Logs**  وشرحها (كيف بتتفعل ووين بتتخزن) مؤجل لمحاضرة جاية.

لما بحكي على مستوى التيبل، بيعتمد على تكنولوجيا عملتها أوراكل من **11g وطالع** يعني بـ10g ما كانت موجودة أصلاً. هاي التكنولوجيا اسمها **Recycle Bin**. فأنا هلأ بقدر أرجع للجدول من خلال أمر **Flashback to Before Drop**.

لما بحكي عن الـTransaction أو أي Query من الـRecycle Bin أو أي عملية ثانية، هاي كلها بتعتمد على حاجة اسمها **Undo Data**. ولهيك، لازم أكون فاهم منيح الـUndo Data والـUndo Management قبل ما أدخل بموضوع الفلاش باك لأنه الـVersion والـQuery والـTable والـTransaction، كلهم بياخذوا بياناتهم من الـUndo Data.

فلازم أكون فاهم كمان شو يعني **Retention Policy** وشو يعني **الضمانة (Guarantee)** لهاي الـRetention وهاد كله كان شرحناه بمحاضرة الأمس عن الـUndo Management (كيف الـUndo Tablespace، وشو الـRetention تبعه).

## 2. Flashback Drop (استرجاع تيبل انحذف)

لما تعمل `DROP TABLE`، أوراكل (من 11g) ما بتمسح الجدول فعلياً بترسله على **Recycle Bin** (زي سلة المحذوفات بالويندوز). فتقدر ترجعه.

### خلفية سريعة قبل ما أبلش
أنا هلأ لازم أكون مسجلة دخول بيوزر عادي (`student`) مش SYS، لأنه **SYS ما بتستخدم Recycle Bin أصلاً** هاي جربتها وشفتها بعيني أول مرة.

الجدول اللي شغلت عليه: `my_test` فيه عمودين `id` و`name`، وفيه صفين: (1, 'Sara') و(2, 'Lina').

### الخطوات اللي طبقتها بالتفصيل

**أ) اعملت جدول تجريبي وملأيته:**
```sql
CREATE TABLE my_test (id NUMBER, name VARCHAR2(50));
INSERT INTO my_test VALUES (1, 'Sara');
INSERT INTO my_test VALUES (2, 'Lina');
COMMIT;
```

**ب) تأكدت من عدد السجلات:**
```sql
SELECT COUNT(*) FROM my_test;   -- طلعت 2
```

**ج) حذفت الجدول:**
```sql
DROP TABLE my_test;
```

**د) جربت أستعلم عنه، طلع لي error:**
```sql
SELECT * FROM my_test;
-- ORA-00942: table or view does not exist
```

**هـ) شفت السلة:**
```sql
SHOW RECYCLEBIN;
```
وطلع سطر فيه:
- **ORIGINAL NAME**: MY_TEST (اسمه الحقيقي)
- **RECYCLEBIN NAME**: اسم غريب طويل زي `BIN$HN+UNxOHREWGXeWAG85IdQ==$0` (فيه رموز خاصة لهيك لازم يتحط بين علامتين اقتباس مزدوجتين)
- Object Type، وقت الحذف

**فهمت من هون:** الجدول ما انمسح فعلياً، هو موجود بس تحت اسم مستعار غريب.

---

### 1️⃣ الطريقة الأولى: أستعلم من اسم الـRecycle Bin مباشرة
هاي أبطأ طريقة، بس بتفيدني أفهم شو صاير بالخفاء.

```sql
SELECT * FROM "BIN$HN+UNxOHREWGXeWAG85IdQ==$0";
```
👈 هيك شفت البيانات (Sara وLina) وهي لسا بالسلة، بدون ما أرجعها كجدول حقيقي بعد.

---

### 2️⃣ الطريقة الثانية: أعمل جدول جديد من بيانات السلة (نسخة)
هاي مو "استرجاع" حقيقي، هاي بس نسخ البيانات لجدول جديد باسم تاني:

```sql
CREATE TABLE my_test_new AS
SELECT * FROM "BIN$HN+UNxOHREWGXeWAG85IdQ==$0";
```

**تأكدت:**
```sql
SELECT * FROM my_test_new;
```
👈 صار عندي جدول جديد اسمه `my_test_new` فيه نفس البيانات. بس خلي بالي: **الجدول الأصلي `my_test` لسا محذوف وبالسلة، ما رجع** هاد بس نسخة إضافية.

---

### 3️⃣ الطريقة الثالثة (الأسهل والأهم): FLASHBACK TABLE الاسترجاع الحقيقي
هاي اللي فعلياً جربتها ونجحت معي:

```sql
FLASHBACK TABLE my_test TO BEFORE DROP;
```

**تأكدت:**
```sql
SELECT * FROM my_test;
```
والنتيجة طلعت فعلاً:
```
ID NAME
1  Sara
2  Lina
```
👈 هيك **الجدول الأصلي رجع بنفس اسمه القديم `my_test`**، مو نسخة جديدة زي الطريقة التانية. هاد الفرق المهم بين الطريقة 2 والطريقة 3:
- **CTAS من الـRecycle Bin** = نسخة جديدة باسم مختلف، الأصلي بضل محذوف
- **FLASHBACK TABLE** = استرجاع حقيقي، الجدول رجع بنفس اسمه واستمراريته

---

### 4️⃣ الطريقة الرابعة: أرجّعه باسم مختلف (RENAME TO)
لو ما بديش أرجعه بنفس الاسم القديم (مثلاً عشان بدي أخليه نسخة احتياطية جنب الأصلي):

```sql
DROP TABLE my_test;

FLASHBACK TABLE my_test TO BEFORE DROP RENAME TO my_test_backup;
```

**تأكدت:**
```sql
SELECT * FROM my_test_backup;   -- رح تطلع فيها البيانات

SELECT * FROM my_test;          -- رح ياخذني error، لأنه القديم بضل محذوف
```
👈 لاحظت هون: لما بستخدم RENAME، **الجدول الأصلي بضل محذوف، والاسترجاع بيصير باسم جديد بس**.

---

### ⚠️ ملاحظة الـGRANTS (لازم أرجعها يدوياً)
بأي طريقة من الطريقتين (2 أو 3 أو 4)، الصلاحيات (Grants) اللي كانت معطاة على الجدول الأصلي **ما بترجع تلقائياً** لازم أعيد أعطيها يدوياً بعد الاسترجاع:
```sql
GRANT SELECT ON my_test TO اسم_اليوزر;
```

## 3. الحذف النهائي PURGE

هلأ هاد الجزء اللي بيوضحلي **ليش PURGE خطير**. جربته على نفس الجدول:

```sql
DROP TABLE my_test PURGE;
```

وجربت أرجعه:
```sql
FLASHBACK TABLE my_test TO BEFORE DROP;
```

وطلع لي بالظبط:
```
ORA-38305: object not in RECYCLE BIN
```

**فهمت من هيك:** كلمة PURGE معناها "امسحه نهائي، حتى لا تحطه بالسلة أصلاً"  فمافي وين أرجعله، لأنه أصلاً ما دخل الـRecycle Bin من الأساس. تماماً زي الفرق بين **Delete العادي** (بيروح عالسلة، بقدر أرجعه) و**Shift+Delete** (بيمسح نهائي، ما في رجعة).

وفيه صيغ تانية لـPURGE لو الجدول أصلاً موجود بالسلة وبدي أفضّيه نهائياً من فيها:
```sql
PURGE TABLE my_test;          -- تفضية جدول واحد بس من السلة
PURGE RECYCLEBIN;              -- تفضية كل السلة تبعتي أنا
PURGE TABLESPACE ts_name;      -- تفضية كل السلة لكل الـTablespace
```

**متى بستخدم PURGE؟** بس لو عندي مشكلة **مساحة فعلية** (الجدول كبير وعم ياخذ حيّز كبير بالـTablespace). غير هيك، ما في داعي بترك الجدول بالسلة براحته، وبحذفه نهائياً بس لو احتجت المساحة فعلاً.

---

### 🎯 خلاصة الجدول (بالمقارنة، عشان تترسخ بمخي):

| العملية | شو بيصير؟ | ممكن أرجعه بعدها؟ |
|---|---|---|
| `DROP TABLE` (عادي) | بيروح عالRecycle Bin | ✅ نعم، بـFlashback |
| `SELECT FROM "BIN$..."` | بس بشوف البيانات وهي بالسلة | |
| `CREATE TABLE AS SELECT FROM "BIN$..."` | بعمل نسخة جديدة | الأصلي بضل بالسلة |
| `FLASHBACK TABLE ... TO BEFORE DROP` | بيرجع الجدول الأصلي بنفس اسمه | ✅ استرجاع حقيقي |
| `FLASHBACK TABLE ... RENAME TO` | بيرجعه باسم جديد | الأصلي بضل محذوف |
| `DROP TABLE ... PURGE` | حذف نهائي، ما بيروح عالسلة | ❌ لأ، ما في رجعة |

## 4. Flashback Version Query (تتبع تغيّر القيمة عبر الزمن)

### شو الفكرة أصلاً؟ (قبل ما أروح على الكود)

لحد هلأ تعلمت كيف أرجّع جدول كامل انحذف. بس هلأ الموضوع مختلف شوي: أنا **مش بدي أرجّع جدول محذوف**، أنا بدي أعرف: **قيمة معينة جوا جدول موجود وشغّال عادي، كيف تغيّرت مع الوقت؟**

يعني مثلاً عمود `name` لصف معين كان قيمته "Version 1"، بعدين صار "Version 2"، بعدين صار "Version 3" بدي أشوف كل هاي المراحل مع بعض بجدول واحد، بدون ما أعمل أي Restore أو Flashback فعلي للجدول.

هاي بتعتمد بالكامل على **الـUndo Data** (نفس الدفتر السري اللي حكينا عنه قبل) مش على الـRecycle Bin.

### الكود اللي طبقته، خطوة خطوة

**الخطوة 1: أعمل الجدول (وهاد المرة زودت عمود تاريخ `dt` كمان):**
```sql
CREATE TABLE local_flash_test (
    id   NUMBER,
    name VARCHAR2(50),
    dt   DATE
);
```

**الخطوة 2: أفعّل خاصية Row Movement خطوة لازم ما أنساها:**
```sql
ALTER TABLE local_flash_test ENABLE ROW MOVEMENT;
```
ليش هاي الخطوة مهمة؟ لأنه الفلاش باك (بأنواعه المرتبطة بحركة الصفوف) بحاجة الجدول يكون "قادر" إنه يتتبع مكان كل صف حتى لو تحرك فيزيائياً جوا الـTablespace. عادة بتكون مفعّلة افتراضياً، بس المهندس شدد إنه "أأكد المؤكد" لأنه لو ما فعّلتها، ممكن الفلاش باك ما يشتغل زي ما المفروض.

**الخطوة 3: أضيف أول قيمة (Version 1):**
```sql
INSERT INTO local_flash_test VALUES (1, 'Version 1', SYSDATE);
COMMIT;
-- هون بستنى دقيقة أو دقيقتين فعلياً، مش بس أكمل عالطول
```
ليش لازم أستنى فعلياً؟ لأنه بدي فرق زمني حقيقي بين كل نسخة، عشان لما أستعلم بعدين، يبين لي بوضوح إنه كل قيمة إلها وقت بداية ونهاية مختلف. لو نفذت كل الأوامر مرة وحدة بسرعة، كل القيم رح يطلع وقتها متقارب جداً ومش رح يبين الفرق منيح.

**الخطوة 4: أعدّلها (Version 2):**
```sql
UPDATE local_flash_test SET name = 'Version 2' WHERE id = 1;
COMMIT;
-- بستنى دقيقة كمان
```

**الخطوة 5: أعدّلها مرة تالتة (Version 3):**
```sql
UPDATE local_flash_test SET name = 'Version 3' WHERE id = 1;
COMMIT;
```

هلأ صار عندي: نفس الصف (id=1) مرّ بثلاث حالات مختلفة خلال ~3-4 دقائق: Version 1 → Version 2 → Version 3. القيمة الحالية بالجدول هلأ هي "Version 3" بس، لو عملت `SELECT` عادي رح أشوف بس هاي القيمة الأخيرة.

**الخطوة 6: الاستعلام السحري أشوف كل النسخ مع بعض:**
```sql
SELECT versions_xid,
       versions_starttime,
       versions_endtime,
       id,
       name
FROM   local_flash_test
VERSIONS BETWEEN TIMESTAMP
       (SYSTIMESTAMP - INTERVAL '3' MINUTE) AND SYSTIMESTAMP
WHERE  id = 1;
```

### أفكك هاد الاستعلام كلمة كلمة عشان يترسخ بمخي

- **`VERSIONS BETWEEN TIMESTAMP (...) AND SYSTIMESTAMP`**: هاد الجزء هو "قلب" الاستعلام بقول لأوراكل "وريني كل النسخ اللي مرّ فيها هاد الصف، من وقت معين لحد هلأ".
- **`SYSTIMESTAMP - INTERVAL '3' MINUTE`**: يعني "الوقت الحالي ناقص 3 دقايق" أي إنه بدي أرجع 3 دقايق للوراء بالضبط. لازم هاد الرقم يكون أكبر من أو يساوي الوقت اللي فعلياً استنيته بين كل UPDATE، وإلا ما رح ألاقي كل النسخ.
- **`SYSTIMESTAMP`**: الوقت الحالي، هون وهسا.
- **`versions_xid`**: رقم الـTransaction (معرّف فريد) اللي عمل هاد التغيير بالضبط يعني لو بدي أعرف "شو باقي التغييرات صارت بنفس هاد الـTransaction"، بقدر أستخدم هاد الرقم لأدور.
- **`versions_starttime`**: الوقت اللي بلشت فيه هاي القيمة تكون سارية (يعني من إيمتى صارت هاي القيمة).
- **`versions_endtime`**: الوقت اللي خلصت فيه صلاحية هاي القيمة (يعني إيمتى تغيّرت لقيمة تانية). ملاحظة: آخر نسخة (Version 3، الحالية) رح يطلع لها `versions_endtime` = **NULL**، لأنها لسا سارية ومافيش وقت انتهاء إلها لحد هلأ.

### شو النتيجة اللي المفروض تطلع لي؟

جدول فيه **3 صفوف**، كل صف بيمثل نسخة:

| versions_xid | versions_starttime | versions_endtime | id | name |
|---|---|---|---|---|
| xid رقم 1 | وقت الـInsert | وقت أول Update | 1 | Version 1 |
| xid رقم 2 | وقت أول Update | وقت ثاني Update | 1 | Version 2 |
| xid رقم 3 | وقت ثاني Update | NULL (لسا سارية) | 1 | Version 3 |

يعني بشوف قصة كاملة لهاد الصف: من وين بلش، وكيف تغيّر، وإيمتى بالضبط.

### ليش هاد الموضوع "كنز" فعلياً؟ (بفكر فيها لحالي)

**1. أنا كـDBA/مراقب أمني**
بقدر أشوف كل شي صار عالداتابيز مين غيّر، إيمتى غيّر، وشو كانت القيمة قبل وبعد. هاد بدون ما أحتاج أفعّل Auditing منفصل، لأنه المعلومة أصلاً موجودة بالـUndo.

**2. تصحيح غلطة منطقية (Logical Error) بسرعة**
تخيل: عملت `UPDATE` ونسيت شرط الـWHERE، أو حطيت `id` غلط، وعبثت ببيانات مش المفروض تتغير. بدل ما أروح أسأل المدير أو المبرمج "شو كانت القيمة قبل؟"، بقدر أرجع أنا بنفسي، أستعلم بـ`VERSIONS BETWEEN`، وأشوف بالضبط شو كانت القيمة الصح قبل غلطتي وأصلحها بنفسي وبسرعة.

بس فيه شرط مهم: يجب أكتشف الغلطة **بسرعة**. ليش؟ لأنه هاد كله جاي من الـUndo، والـUndo إله عمر محدود (Retention Period).

### ⚠️ التحذير المهم: حدود الوقت (Retention Period)

لو حاولت أرجع لفترة أبعد من الـUndo Retention المحدد بالداتابيز (مثلاً الإعدادات محددة Retention بـ5 دقايق بس، وأنا حاولت أرجع 10 دقايق للوراء بـ`INTERVAL '10' MINUTE`)، رح ياخذني error شبيه بـ:
```
ORA-01555: snapshot too old
```
أو ما بيرجع لي نتائج أصلاً لأنه هاي البيانات فعلياً انمسحت من الـUndo، مش موجودة أصلاً أرجعلها.

👈 يعني الدرس المستفاد: كل ما اكتشفت الغلطة أسرع، كل ما زادت فرصتي إني أرجع أصلحها. لو أهملتها لساعات أو أيام، ممكن تضيع الفرصة نهائياً.

### 🎯 خلاصتي بجملتين:
1. **Version Query** = بشوف قصة تغيّر قيمة معينة عبر الوقت، بدون ما أرجّع الجدول فعلياً بس لازم أفعّل `ENABLE ROW MOVEMENT` الأول.
2. الاستعلام بيعتمد كلياً على الـUndo، فإله **نافذة زمنية محدودة** (Retention Period) إذا تجاوزتها، البيانات بتكون انمسحت فعلياً وما رح ألاقيها.

## 5. الحالات اللي الفلاش باك ما بيشتغل فيها

### 1. Views ليش ما بتزبط؟

الفكرة يلي فهمتها: الـView أصلاً مش "جدول فيه بيانات مخزّنة". هو بس **جملة SELECT محفوظة** (المهندس سماها "Physical SQL Statement"). يعني كل مرة أكتب `SELECT * FROM my_view`، أوراكل مش بتجيب بيانات مخزّنة، هي **بتنفذ جملة الـSELECT الأصلية من جديد** كل مرة.

النتيجة المنطقية: بما إنه ما في بيانات فعلياً "مخزّنة" بالـView نفسها (هي بس بتجيب من الجداول الحقيقية اللي وراها)، فمافي شي أعمله Flashback إله لأنه "الفلاش باك" أصلاً معناه ارجع لبيانات قديمة، وهون ما في بيانات قديمة تخص الـView نفسها.

### 2. Fixed Tables ليش ما بتزبط؟

الفكرة: المهندس قال هاي "بتاخذ موضوع الـDDL". يعني لما أعمل `ALTER TABLE` (أغيّر بنية الجدول أضيف عمود، أحذف عمود...)، الفيرجنز ما بتقدر تتبع هيك تغييرات.

السبب الجذري: الفيرجنز بتشتغل **بس** على مستوى **الداتا** اللي بتتخزن بالـUndo Tablespace مش على مستوى **بنية (Structure)** الجدول نفسه. يعني `VERSIONS BETWEEN` بتوريني "القيمة تغيّرت من X لـY"، بس ما بتوريني "العمود اتحذف" أو "اتضاف عمود جديد" هاد شي تاني كلياً (DDL مش DML).

Fixed Tables تحديداً هي جداول داخلية بأوراكل بتعكس حالة الذاكرة اللحظية للنظام (زي `V$` views) مش بيانات "مخزّنة" بالمعنى التقليدي، فمافيها Undo أصلاً.

### 3. External Tables ليش ما بتزبط؟

الفكرة: External Table = بنيته (Structure) موجودة **جوا** الداتابيز (Columns، أنواع البيانات...)، بس **البيانات الفعلية موجودة برا الداتابيز** بملف Excel أو Text عالهارد ديسك.

ليش ما بيزبط الفلاش باك؟ لأنه لو رحت عدّلت الملف نفسه (فتحته بالنوتباد وغيّرت فيه وحفظت)، هاد التعديل صار مباشرة عالـOS (نظام التشغيل)، مش عبر الداتابيز. وبما إنه مش عبر الداتابيز، **مافي Undo ولا Redo** يسجّل هاد التغيير أصلاً. فمافي "دفتر سري" يحتفظلي بالنسخة القديمة.

### 4. Temporary Tables (Global Temporary Table) ليش ما بتزبط؟

الفكرة: الجدول المؤقت بيكون مرتبط بالـSession (الجلسة) بس.

الفرق المهم اللي حكاه المهندس:
- **الجدول العادي**: لما أعمل `COMMIT`، هاد بيأكد الحفظ (البيانات بتضل).
- **الجدول المؤقت**: بالعكس تماماً لما أعمل `COMMIT` (أو أسكر الـSession)، البيانات **بتنمسح** تلقائياً! لأنه المفروض هاد الجدول أستخدمه مؤقتاً بس (أعمل عليه SELECT وUPDATE وحسابات وسيطة)، وبالآخر آخذ النتيجة النهائية وأحطها بجدول عادي.

النتيجة: بما إنه غالباً الجدول المؤقت بيكون فاضي بعد أي Commit، مافي داعي أصلاً أعمل عليه Flashback مافي شي أرجعله.

### 5. تأثير Segment Shrink على الفلاش باك

الفكرة: أحياناً الـDBA بيعمل:
```sql
ALTER TABLE my_table SHRINK SPACE;
```

شو بيصير فيزيائياً: تخيلي البيانات محجوزة بالـTablespace من بلوك 1 لـ50، وبعدين بيانات ثانية محجوزة من 50 لـ100، ومن 100 لـ120... لو حذفت البيانات اللي بين 50 و100، المكان بيضل فاضي ما بيلتصق تلقائياً مع باقي البيانات (زي "ثقوب" وسط الملف).

الـSHRINK بياخذ كل البيانات ويعيد ترتيبها جنب بعضها بدون فراغات، عشان يوفر مساحة.

العلاقة بالفلاش باك: بما إنه هاي العملية بتحرّك موقع البيانات فيزيائياً جوا الملفات، ممكن تأثر سلباً على قدرتي إني أرجع بالفلاش باك لفترات قبل عملية الـShrink يعني ممكن ألاقي البيانات وممكن ما ألاقيها
