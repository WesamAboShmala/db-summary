## 🎯 1. المصادقة مقابل التفويض (Authentication vs Authorization)

* **المصادقة (Authentication):** هي عملية التأكد من الهوية (هل أنت فعلاً الشخص الذي تدّعيه؟).
* **التفويض (Authorization):** هو تحديد ما يحق للمستخدم القيام به من صلاحيات وعمليات داخل الداتابيز بعد نجاح عملية الدخول.

---

## 💻 2. الدخول بدون باسورد (OS Authentication)

عند نسيان كلمة مرور مستخدم عالي الصلاحيات مثل `SYSTEM` أو `SYS`، يمكنك فتح الـ SQL*Plus وكتابة الأمر التالي مباشرة:

```sql
CONNECT / AS SYSDBA

```

* **ماذا تعني الـ `/`؟** تعني الدخول باستخدام **Windows OS Authentication**، حيث تمنح المستخدم أعلى صلاحية للتحكم بالقاعدة بناءً على مصادقة نظام التشغيل.
* **تعديل الباسورد بعد الدخول:** بمجرد الدخول، يمكن تغيير كلمة مرور أي مستخدم (مثل SYSTEM) باستخدام الأمر:

```sql
ALTER USER system IDENTIFIED BY new_password;

```

> ⚠️ **شرط نجاح هذه العملية:** يجب أن يكون المستخدم الحالي على نظام التشغيل (OS) يمتلك صلاحيات **Administrator** وعضواً في مجموعة **ORA_DBA** داخل الـ (Local Users and Groups).

> 💡 **ملاحظة مهمة //** لنجاح أي اتصال بقاعدة البيانات، يحتاج المستخدم بشكل عام إلى:
> `(IP, SID, Port) + Authentication + Privileges`

## 🛠️ 3. إنشاء Database User (الطريقة الاحترافية)

يتم إنشاء مستخدم ومنحه الصلاحيات الأساسية عبر الأوامر التالية:

```sql
CREATE USER store_admin IDENTIFIED BY store;
GRANT CREATE SESSION TO store_admin;
GRANT RESOURCE TO store_admin;

-- أو إعطاؤه صلاحيات كاملة مبدئياً:
GRANT DBA TO store_admin;

```

### 💻 طريقة المبرمجين (جدول التطبيق الخلفي):

هذه هي الطريقة التي يراها أغلب المبرمجين الأنجح لإدارة المستخدمين من داخل التطبيق:

```sql
CREATE TABLE users_tb (
  user_id   NUMBER,
  password  VARCHAR2(150), -- لازم تكون مشفرة (Hashed)
  status    NUMBER,
  cr_date   DATE
);

```

**عملية التحقق عند تسجيل الدخول:**

```sql
SELECT COUNT(*) FROM users_tb
WHERE user_id   = :username
  AND password  = ENCRYPT(:entered_password)
  AND status    = 1;

```

* إذا الناتج = `1` ← ✅ دخول ناجح.
* إذا الناتج = `0` ← ❌ فشل الدخول.

> 🚨 **تنبيه مهم:** الباسورد في الجدول يجب تخزينها كـ **Hash مشفر**، وليس كنص واضح (Plain Text).

### 🧐 رأي المهندس في الطريقتين:

هذه الطريقة لا يمكن القول عنها خطأ أو صح بشكل مطلق، وهناك من يتساءل: لماذا أُعقد نفسي وأقوم بإنشاء مستخدم كامل في الداتابيز كـ Object كبير بينما يمكنني الاكتفاء بإنشاء يوزر كـ Record (سجل) داخل جدول؟

```sql
-- عند إنشاء مستخدم كـ Object كامل، يمكنك حذفه ومحتوياته هكذا:
DROP USER T11 CASCADE;

```

**خلاصة رأي المهندس:** كل طريقة لها استخدامها الخاص، ولكل واحدة مزايا وعيوب. دورك كمهندس هو دراسة الموقف وتحديد الخيار الأسهل والأنسب للتنفيذ في مشروعك.

## 📁 4. مفهوم الـ Tablespace وعزل البيانات

**الـ Tablespace:** هو ببساطة الإطار المنطقي (Logical Framework) لتخزين البيانات في أوراكل.
بدلاً من وضع كل بيانات المستخدمين في مكان واحد (المساحة الافتراضية التي تسمى `USERS`)، نقوم بإنشاء مساحات خاصة لكل تطبيق أو مستخدم لتنظيم وعزل العمل.

```sql
-- إنشاء Tablespace جديد
CREATE TABLESPACE ucas_tbs
  DATAFILE 'ucas.dbf' SIZE 10G;

-- ربط المستخدم بـ Tablespace معين وتحديد كوتة المساحة
CREATE USER Ahmed IDENTIFIED BY pass123
  DEFAULT TABLESPACE ucas_tbs
  QUOTA 100M ON ucas_tbs;

```

### ⚙️ الأوامر الأساسية لإدارة الحسابات:

```sql
-- تغيير الباسورد
ALTER USER Ahmed IDENTIFIED BY new_password;

-- قفل الحساب
ALTER USER Ahmed ACCOUNT LOCK;

-- فتح الحساب
ALTER USER Ahmed ACCOUNT UNLOCK;

-- حذف المستخدم بكل محتوياته
DROP USER Ahmed CASCADE;

```

## 👥 5. أنواع الـ Authentication Methods الثلاثة

### 1. Password Authentication

هي الطريقة الأكثر انتشاراً، حيث يكون لكل مستخدم معرف (ID) وكلمة مرور خاصة به مخزنة في قاعدة البيانات.

> 🔄 **تطوير حديث:** بعض التطبيقات الحديثة تستغني عن كلمات المرور التقليدية وتستخدم بصمة الإصبع أو بصمة الوجه كبديل رقمي للتحقق من الهوية (Identification) وهي تؤدي نفس الغرض.

```sql
CREATE USER ahmed IDENTIFIED BY mypassword;

```

### 2. External Authentication (OS Authentication)

تعتمد هذه الطريقة على مفهوم **الدخول الموحد (Single Sign-On - SSO)**. إذا قام المستخدم بتسجيل دخوله لمرة واحدة على نظام التشغيل (Windows) أو الدومين (Domain)، فإن قاعدة بيانات أوراكل تتعرف عليه تلقائياً وتسمح له بالدخول دون مطالبته بكلمة مرور مرة أخرى.

```sql
-- تفعيل الميزة في باراميتر فايل
ALTER SYSTEM SET remote_os_authent = FALSE SCOPE = SPFILE;
ALTER SYSTEM SET os_authent_prefix = 'OPS$' SCOPE = SPFILE;

-- ثم إعادة تشغيل الداتابيز لتطبيق التغييرات
STARTUP FORCE;

-- إنشاء اليوزر الخارجي
CREATE USER ops$windows_username IDENTIFIED EXTERNALLY;

-- الدخول بدون باسورد
CONNECT /

```

> 💡 **الفكرة:** اليوزر يسجل دخوله على الـ OS مرة واحدة، وبعدها يتحرك على كل البرامج والداتابيز بسلاسة (SSO).

### 3. Global Authentication

نفس فكرة الـ External تماماً، ولكنها مخصصة للأنظمة الموزعة **(Distributed Database)** التي تحتوي على أكثر من قاعدة بيانات في مواقع جغرافية مختلفة. تستخدم هذه الطريقة بروتوكول **LDAP** لربط مستخدم الدومين مع Oracle Domain.

> 🏢 **مثال:** بنك فلسطين لديه قاعدة بيانات في كل فرع؛ المدير يستطيع الدخول بمستخدم واحد موحد على جميع القواعد في كل الفروع.

## ⚙️ 6. باراميترات المصادقة الخارجية (OS Authentication Parameters)

```sql
-- عرض البراميترات المتعلقة بالـ OS داخل الداتابيز
SHOW PARAMETER os

```

* `remote_os_authent` ← القيمة الافتراضية لها هي `FALSE`.
* `os_authent_prefix` ← البادئة (Prefix) التي يجب أن يبدأ بها اسم اليوزر الخارجي.

### 🔴 ثغرة باراميتر REMOTE_OS_AUTHENT الخطيرة:

هذا الباراميتر مسؤول عن السماح بالولوج إلى الداتابيز باستخدام مصادقة نظام التشغيل **عن بُعد (Remote)**. لتفعيله، يتم تعديله بالأمر التالي:

```sql
ALTER SYSTEM SET remote_os_authent = TRUE SCOPE=SPFILE;
-- ملاحظة: لن يظهر التعديل إلا بعد إعادة التشغيل:
STARTUP FORCE;

```

> 🛑 **لماذا هي ثغرة خطيرة جداً؟** لأن تحويل القيمة إلى `TRUE` يعني أن الداتابيز تثق عمياء في نظام التشغيل الخاص بالجهاز "البعيد" المتصل عبر الشبكة. لو قام أي شخص غير مصرح له بربط لابتوبه الشخصي بالشبكة، وأنشأ مستخدم Admin على جهازه بنفس الاسم المطلوب، ستسمح له الداتابيز بالدخول فوراً دون طلب باسورد!
> 🛡️ **موقف أوراكل:** بسبب هذه الخطورة، قامت شركة أوراكل منذ إصدار `11g` باحتساب هذا الباراميتر كـ **Deprecated** (غير موصى به)، وفي الإصدارات الحديثة مثل `19c` و `21c` **تم إلغاؤه تماماً** ولم يعد يعمل لحماية الشركات من هذا النوع من الاختراق.

### ⚙️ باراميتر OS_AUTHENT_PREFIX:

أي مستخدم يتم إنشاؤه ليعمل بالمصادقة الخارجية يجب أن يبدأ بالبادئة المحددة هنا. القيمة الافتراضية هي `OPS$`، ولكن قام المهندس بتغييرها إلى `UCAS$` كالتالي:

```sql
ALTER SYSTEM SET os_authent_prefix = 'UCAS$' SCOPE=SPFILE;

```

**المختصر لإنشاء ودخول اليوزر الخارجي:**

```sql
CREATE USER "ucas$Ahmed" IDENTIFIED EXTERNALLY;

```

> 🔍 **لمعرفة اسم المستخدم الصحيح لجهازك:** افتح الـ CMD واكتب الأمر `whoami`؛ خذ الاسم الناتج واستبدله بـ `Ahmed` ليتصل جهازك بالداتابيز بدون باسورد.

## 👥 8. الملفات الشخصية وإدارة الموارد (User Profiles & Resource Control)

**البروفايل (Profile):** هو كائن (Object) يتم إنشاؤه للتحكم في مسارين أساسيين للمستخدم. الهدف الجوهري منه هو عكس **سياسة أمن المعلومات (Security Policy)** المقرة في المؤسسة وتحويلها إلى أوامر تقنية داخل قاعدة البيانات.

### 📊 المسار الأول: إدارة الموارد (Resource Control)

يتحكم البروفايل في استهلاك المستخدم لكل من:

* كمية استهلاك المعالج والـ CPU.
* الـ **Idle Time** (الوقت الذي يقضيه المستخدم دون عمل وهو متصل).
* عمليات القراءة والكتابة (Read/Write) على الهارد ديسك.
* المساحة المتاحة للمستخدم في الذاكرة العشوائية SGA.

> 👥 **أنواع المستخدمين حسب الموارد:**
> 1. **يوزر الـ Parent (Owner):** وهو الحساب المالك لجميع الـ Objects والجداول؛ وبما أنه الأكثر عملاً على الداتابيز، يتم وضع معايير (Criteria) مرنة وخاصة له في الـ Resources.
> 2. **يوزر عادي (Read/Write فقط):** حساب يقع تحت سيطرتك الكاملة، تحدد له بدقة كم يقرأ وكم يكتب.
> 
> 

### 🔑 المسار الثاني: سياسة كلمات المرور (Password Control)

يتحكم البروفايل في خصائص الأمان التالية للكلمات المرور:

* **Password History:** كم مرة يمكن الاحتفاظ وإعادة استخدام الباسورد القديمة.
* **Complexity Verification:** التحقق من مدى تعقيد وقوة كلمة المرور.
* **Expiration:** متى تنتهي صلاحية الباسورد ويجب تغييرها.
* **Account Lock:** بعد كم محاولة خاطئة يتم إغلاق الحساب تلقائياً.

### 🔄 تسلسل تحويل السياسة:

```
سياسة المؤسسة (Policy) 👈 تحويلها برمجياً لبروفايل (Profile) 👈 ربط وتعيين البروفايل للمستخدم (Assign to User)

```

* **مثال:** تطلب سياسة أمن المؤسسة احتواء الباسورد على حرف كبير + رمز خاص + أرقام. يقوم الـ DBA بعمل Enforce (فرض) على كل المستخدمين لتكون كلمات مرورهم Complex عبر البروفايل.

```sql
-- عرض جميع المستخدمين في قاعدة البيانات
SELECT username FROM dba_users;

-- معرفة اسم البروفايل الخاص بمستخدم معين بالتحديد
SELECT profile FROM dba_users WHERE username = UPPER('UCAS$desktop-sl0qutr\click');

```

## 📊 9. أقسام قاموس البيانات الثلاثة (Data Dictionary Levels)

تنقسم جداول قاموس البيانات في أوراكل إلى 3 مستويات أساسية:

1. **DBA:** يعرض جميع الكائنات (Objects) الموجودة في قاعدة البيانات بالكامل بغض النظر عن المالك. لا يمكن لأي يوزر الاستعلام منه إلا إذا امتلك صلاحية الـ DBA؛ وإذا سُحبت منه يظهر له خطأ "الجدول غير موجود".

```sql
SELECT COUNT(*) FROM dba_objects;

```

2. **USER:** يعرض فقط الكائنات التي يملكها المستخدم الحالي نفسه الذي قام بالاتصال (جدول شخصي وخاص بالـ Owner).

```sql
SELECT COUNT(*) FROM user_objects;

```

3. **ALL:** يعرض كل الكائنات التي يمتلك المستخدم الحالي صلاحية الوصول إليها والاستعلام منها، حتى لو لم يكن هو المالك الأصلي لها.

```sql
SELECT COUNT(*) FROM all_objects;

```

> 🧪 **تجربة عملية إثبات الصلاحيات:**
> عند الدخول بحساب الـ `SYS` ومنح صلاحية الـ DBA للمستخدم، سيعرض له البروفايل والبيانات تماماً. وعند سحبها عبر أمر الحذف، لن يعود قادراً على رؤية شيء.

```sql
-- سحب صلاحية الـ DBA من مستخدم
REVOKE dba FROM "UCAS$DESKTOP-SL0QUTR\CLICK";

-- إعادة صلاحية الـ DBA للمستخدم
GRANT dba TO "UCAS$DESKTOP-SL0QUTR\CLICK";

```

## 🔑 10. تعديل البروفايل وحظر الحسابات (Password Control in Action)

```sql
-- عرض تفاصيل ومحددات الموارد داخل البروفايل الافتراضي DEFAULT
SELECT resource_name, limit FROM dba_profiles WHERE profile = 'DEFAULT';

```

من المخرجات نلاحظ مثلاً: `FAILED_LOGIN_ATTEMPTS = 10` (عدد المحاولات الخاطئة المسموحة) و `PASSWORD_LOCK_TIME = 1` (مدة قفل الحساب باليوم).

### 🛠️ إنشاء بروفايل مخصص وتعديل المحاولات إلى 5:

```sql
-- إنشاء البروفايل المخصص وضبط المحاولات الفاشلة بـ 5
CREATE PROFILE ucas_profile LIMIT FAILED_LOGIN_ATTEMPTS 5;

-- ربط البروفايل الجديد بالمستخدم الخاص بنا
ALTER USER "UCAS$DESKTOP-SL0QUTR\CLICK" PROFILE ucas_profile;

```

> 🧪 **تجربة الحظر:** عند محاولة الدخول 5 مرات بشكل خاطئ، سيقوم النظام تلقائياً بقفل الحساب وتحويل حالته إلى Locked. لفك القفل عن الحساب نستخدم أمر الـ Unlock الذي درسناه سابقاً:

```sql
ALTER USER WESAM ACCOUNT UNLOCK;

```

### ⏳ انتهاء صلاحية كلمة المرور (Password Expiration):

عند تحديد صلاحية الباسورد بـ 180 يوماً (أي حوالي 6 أشهر)، فبمجرد انتهاء هذه المدة تظهر الداتابيز تلقائياً رسالة تطلب تغيير الباسورد. يبدأ النظام بإرسال تحذيرات للمستخدم قبل أسبوع تقريباً من تاريخ الانتهاء الفعلي.

## 🛡️ 11. استراتيجيات الحماية والتحذيرات الأمنية المتقدمة

### 🚨 1. خطر قفل الحساب وهجمات حجب الخدمة (DoS Attack):

المبرمج العادي يظن أن قفل الحساب بعد 5 محاولات فاشلة يحمي النظام، لكن **المهندس ينبهنا أن هذا خطر جداً!**
إذا عرف المهاجم (Hacker) اسم مستخدم موظف رئيسي في السيستم، سيقوم عمداً بإدخال باسوورد خاطئة 5 مرات متتالية في أقل من دقيقة. النظام سيقوم فوراً بقفل حساب الموظف الشرعي! هنا يكون الهكر قد نجح في قطع الخدمة وتعطيل عمل الشركة دون الحاجة لاختراق أي شيء، وهذا ما يسمى بهجوم حجب الخدمة **(Denial of Service - DoS)**.

#### 🛠️ الحلول المقترحة من المهندس:

* **حظر الـ IP وليس اليوزر:** بدلاً من قفل الحساب كلياً، نقوم بمراقبة الجهاز الذي تصدر منه المحاولات الفاشلة ونحظر رقم الـ IP الخاص به (عبر الفاير وول أو الداتابيز)، وبذلك يستمر حساب الموظف بالعمل بشكل طبيعي.
* **نظام التنبيهات الفورية:** ربط النظام بآلية ترسل رسائل نصية (SMS) أو تنبيهات فورية للمسؤول (Administrator) تفيد بوجود محاولات تخمين جارية على الحساب، ليتحرك فريق الأمن السيبراني فوراً قبل حدوث القفل.

### ⚙️ 2. ميزة أوراكل الجاهزة لتعقيد كلمة المرور:

توفر أوراكل دالة جاهزة ومبرمجة داخلياً (Built-in) لفحص شروط تعقيد كلمات المرور تسمى **`VERIFY_FUNCTION_11G`**.

> 💬 **كلام المهندس:** "تخيل لو أنا بدي أبرمج شروط فحص الباسورد (هل تحتوي رموز؟ هل هي نفس الاسم؟ هل هي مقلوبة؟) كم سآخذ من الوقت لكتابة آلاف الأسطر البرمجية؟ الكوماند في أوراكل بسيط جداً وهذه الميزة بمجرد ربطها بالبروفايل تتولى فحص التعقيد تلقائياً وترجع True أو False وترفض الباسورد الضعيفة فوراً."

### 🚪 3. تأمين الحسابات الافتراضية والـ الباك دور (Back Doors):

عند تثبيت أوراكل، تنزل معها تلقائياً حسابات تجريبية وافتراضية (مثل يوزر `SCOTT` وباسوورده الشهيرة عالمياً `tiger`).

> ⚠️ **تحذير المهندس:** يحذر المهندس بشدة من ترك هذه الحسابات مفتوحة لأن أي هكر مبتدئ يعرفها جيداً وقد يدخل منها كـ "باب خلفي" لتدمير النظام. الحل الأمثل هو قفلها فوراً (`ACCOUNT LOCK`) أو تغيير كلماتها الافتراضية.

### 🚫 4. حظر حذف المستخدم المتصل:

لا يمكن نهائياً وقطعيّاً عمل `DROP` (حذف) لأي مستخدم طالما أنه يمتلك جلسة نشطة ومتصل حالياً بقاعدة البيانات (`Currently Connected`).

### 🔑 5. تشفير الباسوورد أثناء النقل والتخزين (Anti-Sniffing):

> 💬 **ماذا قال المهندس حرفياً؟** "الباسورد بتنتقل من الديسك توب أو من الويب أبلكيشن للداتابيز إنكربتد، تستخدم الـ (AES) إنكربتد ألغوريثم وبتتشفر جوا الداتابيز... مطلوب منك تشفر في الإرسال وتشفر في التخزين على أساس أي (Sniffing) ما تظبط معك."

تحمي أوراكل البيانات باستخدام خوارزمية التشفير العالمية المتطورة **AES**. الباسوورد لا تمر في الشبكة كنص واضح ولا تُخزن كنص واضح؛ حتى لو حاول هكر التجسس وسرقة حزم البيانات من الشبكة (**Sniffing**)، أو حاول مدير النظام نفسه قراءة جدول الباسووردات فلن يستطيع فهمها لأنها مشفرة بالكامل.
