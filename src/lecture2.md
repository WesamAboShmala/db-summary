## ❓ 1. ما هو الـ Authorization؟

**الـ Authorization (التفويض):** هو العملية التي تحدد الصلاحيات والعمليات الفعلية التي يسمح للمستخدم بالقيام بها داخل قاعدة البيانات بعد أن يتم التحقق من هويته بنجاح عبر عملية الـ **Authentication**.

> 💡 **بمعنى بسيط جداً:**
> * **Authentication:** التحقق من الهوية 👈 *(من أنت؟)*
> * **Authorization:** تحديد الصلاحيات 👈 *(ماذا يحق لك أن تفعل بعد الدخول؟)*

### 🔄 جدول مقارنة سريع:

| الخاصية | المصادقة (Authentication) | التفويض (Authorization) |
| :--- | :--- | :--- |
| **الهدف** | التحقق من هوية المستخدم | تحديد وتفويض صلاحيات المستخدم |
| **التوقيت** | يحدث أولاً بمجرد محاولة الاتصال بالـ DB | يحدث مباشرة بعد نجاح عملية الاتصال بالـ DB |
| **مثال عملي** | إدخال اسم المستخدم وكلمة المرور | هل يمتلك صلاحية `SELECT` أو `CREATE TABLE`؟ |

---

## 🎯 2. لماذا يعتبر الـ Authorization مهماً؟

* **عدم تماثل الصلاحيات:** ليس من المنطقي أو الآمن أن يمتلك جميع المستخدمين في المؤسسة نفس الصلاحيات المطلقة.
* **حماية البيانات الحساسة:** توجد بيانات وعمليات حساسة جداً يجب حظرها عن العامة (مثل شاشات تعديل ورفع الرواتب).
* **على مستوى قاعدة البيانات (Database Level):** يحدد بدقة مَن يمتلك صلاحية إنشاء الجداول ومَن يحق له القراءة فقط.
* **على مستوى التطبيق (Application Level):** يحدد أي الأزرار والشاشات تظهر لكل موظف بناءً على مسمّاه الوظيفي.

> 🏢 **مثال عملي:**
> * **موظف القبول والتسجيل:** يشاهد شاشات تسجيل الطلاب وقبولهم فقط.
> * **موظف المالية:** يشاهد شاشات الرواتب، المدفوعات، والرسوم الجامعية فقط.
> * كل موظف يعمل بدقة داخل نطاق صلاحياته المحددة دون تداخل.

## 🗂️ 3. أنواع الـ Authorization في Oracle

تنقسم الصلاحيات في أوراكل إلى ثلاثة أنواع أساسية:

### أ. صلاحيات النظام (System Privileges)
هي صلاحيات إدارية عامة في قاعدة البيانات، ولا ترتبط بكائن (Object) معين بذاته، بل تمنح المستخدم القدرة على أداء مهام هيكلية وإدارية داخل الـ DB.

**أمثلة عليها:**
* `CREATE TABLE`: إنشاء جدول داخل السكيمة الخاصة بالمستخدم نفسه.
* `CREATE ANY TABLE`: إنشاء جدول داخل أي سكيمة موجودة في قاعدة البيانات.
* `CREATE USER`: إنشاء مستخدمين جدد.
* `DROP TABLE`: حذف الجداول.
* `ALTER TABLE`: تعديل هيكلية الجداول.
* `CREATE SEQUENCE`, `CREATE VIEW`, `CREATE PROCEDURE`.

#### 🔄 الفرق بين `CREATE TABLE` و `CREATE ANY TABLE`:

| الصلاحية | `CREATE TABLE` | `CREATE ANY TABLE` |
| :--- | :--- | :--- |
| **النطاق** | ينشئ جدولاً داخل سكيمته الخاصة فقط | ينشئ جدولاً داخل أي سكيمة أخرى في الداتابيز |
| **مثال كود** | `CREATE TABLE test_tb (...);` | `CREATE TABLE HR.test_tb (...);` |

---

### ب. صلاحيات الكائنات (Object Privileges)
هي الصلاحيات المرتبطة مباشرة بكائن (Object) معين ومحدد داخل قاعدة البيانات (مثل جدول، دالة، أو تتابع رقمي).

**أمثلة عليها:**
* `SELECT ON table_name`: صلاحية القراءة من جدول معين.
* `INSERT ON table_name`: صلاحية إضافة بيانات جديدة لجدول معين.
* `UPDATE ON table_name`: صلاحية تعديل البيانات داخل جدول معين.
* `DELETE ON table_name`: صلاحية حذف السجلات من جدول معين.
* `EXECUTE ON function/procedure`: صلاحية تشغيل وتنفيذ دالة أو إجراء معين.
* `SELECT ON sequence_name`: صلاحية استخدام تتابع رقمي (Sequence) معين.

---

### ج. التفويض المبني على الأدوار (Role-Based Authorization)

## 📜 4. منح وسحب الصلاحيات (GRANT & REVOKE)

تعتبر جمل المنح والسحب من أهم أوامر لغة التحكم بالبيانات **DCL (Data Control Language)**:
* تمنح الصلاحيات باستخدام الأمر: **`GRANT`**
* تسحب الصلاحيات باستخدام الأمر: **`REVOKE`**

### 🛠️ أولاً: التعامل مع صلاحيات النظام (System Privileges)

```sql
-- منح صلاحية نظام عادية لمستخدم
GRANT CREATE TABLE TO ashaat;

-- منح صلاحية نظام مع إعطائه الحق في تمريرها ومنحها لمستخدمين آخرين
GRANT CREATE TABLE TO ashaat WITH ADMIN OPTION;

-- سحب صلاحية النظام من المستخدم
REVOKE CREATE TABLE FROM ashaat;

```

### 📦 ثانياً: التعامل مع صلاحيات الكائنات (Object Privileges)

```sql
-- منح صلاحية كائن (القراءة من جدول محدد يملكه حساب آخر)
GRANT SELECT ON sys.test_tb TO ashaat;

-- منح صلاحية كائن مع إعطائه الحق في تمريرها ومنحها لغيره
GRANT SELECT ON sys.test_tb TO ashaat WITH GRANT OPTION;

-- سحب صلاحية الكائن من المستخدم
REVOKE SELECT ON sys.test_tb FROM ashaat;

```

## ⚠️ 5. الفرق الجوهري بين System و Object Privilege في السحب

يعتبر هذا الفرق من أهم المفاهيم المتقدمة في إدارة قواعد البيانات، ويظهر بوضوح عند سحب الصلاحيات الممررة:

### 🛠️ في حالة صلاحيات النظام (`System Privilege`):

تستخدم معها الخاصية `WITH ADMIN OPTION`. إذا قام الـ `DBA` بمنح صلاحية لـ `ALI`، وقام `ALI` بتمريرها ومنحها لـ `SAAD`؛ فإنه عند قيام الـ `DBA` بسحب الصلاحية من `ALI` **فإن الصلاحية تبقى مع `SAAD` بشكل طبيعي** ولا تنحسب منه تلقائياً.

### 📦 في حالة صلاحيات الكائنات (`Object Privilege`):

تستخدم معها الخاصية `WITH GRANT OPTION`. إذا قام الـ `DBA` بمنح صلاحية لـ `ALI` على جدول محدد، وقام `ALI` بتمريرها ومنحها لـ `SAAD`؛ فإنه بمجرد قيام الـ `DBA` بسحب الصلاحية من `ALI` **فإن الصلاحية تُسحب من `SAAD` تلقائياً وفوراً!**

> 🛑 **سلسلة الصلاحيات (Chain of Privileges):**
> في صلاحيات الكائنات، إذا قطعت حلقة واحدة في السلسلة، انقطع وانحجب كل ما بعدها تلقائياً لحماية خصوصية وأمن الكائن المخزن.

## 🎭 6. الأدوار (Roles)

**الـ Role (الدور):** هو عبارة عن حزمة (Package) تجمع مجموعة كبيرة من صلاحيات النظام وصلاحيات الكائنات معاً تحت اسم واحد، ليتم منحها للمستخدمين دفعة واحدة.

### ❓ لماذا وجدت الـ Roles؟

بدلاً من تضييع الوقت والجهد في منح عشرات الصلاحيات المنفصلة لكل موظف جديد بشكل يدوي، نقوم بإنشاء `Role` واحدة تحتوي على الصلاحيات المطلوبة للوظيفة، ونمنح هذه الـ `Role` للموظف بكلمة واحدة.

> 💼 **مثال مالي:**
> ننشئ دوراً ماليّاً باسم `Finance_Role` يحتوي على:
> * `SELECT` على 5 جداول مالية.
> * `UPDATE` على 10 جداول حسابات.
> * `EXECUTE` على الدوال المالية.
> * عند قدوم موظف جديد للمالية: `GRANT Finance_Role TO new_employee;`
> * عند انتقال الموظف أو استقالته: `REVOKE Finance_Role FROM old_employee;`
> 
> 

### 🌟 فوائد الـ Roles:

1. **إدارة أسهل للصلاحيات (Easier privilege management):** التعامل مع حزم مجمعة بدلاً من صلاحيات منفردة.
2. **الإدارة الديناميكية (Dynamic privilege management):** لو أضفت جدولاً جديداً للمؤسسة، يكفي إضافة صلاحيته للـ `Role` فقط، وستطبق تلقائياً على كل مستخدم يمتلك هذا الدور.
3. **التوفر الانتقائي للصلاحيات (Selective availability).**

### 💻 خطوات إنشاء واستخدام الـ Role عمليّاً:

```sql
-- 1. إنشاء الـ Role الجديد
CREATE ROLE first_rl;

-- 2. منح صلاحيات نظام للـ Role
GRANT CREATE TABLE TO first_rl;

-- 3. منح صلاحيات كائنات للـ Role
GRANT SELECT ON priv_test_tb TO first_rl;

-- 4. منح الـ Role بالكامل للمستخدم المستهدف
GRANT first_rl TO ashaat;

```

## 🏛️ 7. الأدوار الجاهزة مسبقاً في أوراكل (Predefined Roles)

تأتي بيئة أوراكل مزودة بأدوار مبنية ومجهزة داخل النظام تلقائياً، ومن أشهرها:

| اسم الـ Role الجاهز | الصلاحيات التي يحتوي عليها |
| --- | --- |
| **`CONNECT`** | يمتلك صلاحية اتصال أساسية فقط `CREATE SESSION` |
| **`RESOURCE`** | يمتلك صلاحيات الإنشاء الأساسية للمطورين مثل `CREATE TABLE`, `CREATE PROCEDURE`, `CREATE SEQUENCE`, `CREATE TRIGGER` وغيرها |
| **`DBA`** | يمتلك معظم صلاحيات النظام الإدارية المطلقة (لا تُمنح إلا لمديري النظام) |
| **`SELECT_CATALOG_ROLE`** | يمتلك أكثر من 1700 صلاحية كائن للقراءة والاستعلام من قاموس البيانات (Data Dictionary) |
| **`SCHEDULER_ADMIN`** | يمتلك صلاحيات إدارة الـ Jobs والـ Scheduler والجدولة المؤتمتة |

> 💡 **ملاحظة:** دور الـ `DBA` الذي كنا نستخدمه في المحاضرات السابقة للدخول بأعلى الصلاحيات هو عبارة عن `Predefined Role` جاهز داخل نظام أوراكل.

## 🔒 8. الأدوار المحمية بكلمة مرور (Secure Roles)

**الـ Secure Role:** هو دور عادي جداً، ولكنه محمي بكلمة مرور مخصصة. هذا يعني أن مجرد منح الـ `Role` للمستخدم لا يكفي ليعمل بها؛ بل يجب على المستخدم إدخال الباسورد الصحيحة لتفعيل الدور والاستفادة من صلاحياته عند بدء جلسة العمل.

* **متى نستخدمها؟** تُستخدم للصلاحيات والعمليات الشديدة الحساسية (مثل تعديل الرواتب أو بونص الموظفين)؛ لحماية النظام في حال ترك الموظف جهازه مفتوحاً وغادر مكتبه.

### 💻 كيف تنشئ وتفعّل الـ Secure Role:

```sql
-- 1. إنشاء الـ Secure Role وحمايته بباسورد
CREATE ROLE ucas_sec_role IDENTIFIED BY ucaspw;

-- 2. منح صلاحية كائن للـ Role المحمي
GRANT SELECT ON sec_role_test TO ucas_sec_role;

-- 3. منح الـ Role للمستخدم
GRANT ucas_sec_role TO ashaat;

```

#### 🔄 آلية التفعيل والدخول عند المستخدم:

```sql
-- إذا حاول المستخدم القراءة مباشرة بدون تفعيل الدور بالباسورد 👈 سيظهر له خطأ:
SELECT * FROM sys.sec_role_test;
-- ORA-00942: table or view does not exist

-- الطريقة الصحيحة لتفعيل الدور عبر إدخال الباسورد:
SET ROLE ucas_sec_role IDENTIFIED BY ucaspw;

-- بعد التفعيل بنجاح 👈 تعمل جملة الاستعلام بكفاءة:
SELECT * FROM sys.sec_role_test;

```

### 🔄 الفرق بين Role عادية و Secure Role:

| الخاصية | Role عادية | Secure Role |
| --- | --- | --- |
| **حالة التفعيل** | بمجرد منحها للمستخدم تكون فعالة ونشطة تلقائياً | لا تعمل إلا بعد تفعيلها يدوياً داخل الجلسة باستخدام الباسورد |
| **كود الإنشاء** | `CREATE ROLE role_name;` | `CREATE ROLE role_name IDENTIFIED BY password;` |
| **الاستخدام الأمثل** | مناسبة للصلاحيات اليومية والعادية | مناسبة جداً للعمليات المالية والحساسة لمنع الاختراق الداخلي |

## 📊 9. جداول قاموس البيانات للاستعلام عن الصلاحيات

يمكنك الاستعلام ومراقبة كافة الصلاحيات والأدوار الممنوحة في النظام عبر الجداول التالية:

### 🛠️ أ. جداول صلاحيات النظام (System Privileges Views):

```sql
-- لعرض كل صلاحيات النظام الممنوحة لمستخدم معين في كامل الـ DB (لـ DBA)
SELECT * FROM DBA_SYS_PRIVS WHERE GRANTEE = 'ASHAAT';

-- لعرض صلاحيات النظام التي يمتلكها المستخدم الحالي المتصل الآن
SELECT * FROM USER_SYS_PRIVS;

-- لعرض صلاحيات النظام الممنوحة والمربوطة بالأدوار (Roles)
SELECT * FROM ROLE_SYS_PRIVS;

```

### 📦 ب. جداول صلاحيات الكائنات (Object Privileges Views):

```sql
-- لعرض صلاحيات الكائنات الممنوحة في كامل قاعدة البيانات
SELECT OWNER, TABLE_NAME, PRIVILEGE FROM DBA_TAB_PRIVS WHERE GRANTEE = 'TEST';

-- لعرض صلاحيات الكائنات الممنوحة للمستخدم الحالي
SELECT OWNER, TABLE_NAME, PRIVILEGE FROM USER_TAB_PRIVS WHERE GRANTEE = 'TEST';

-- لعرض صلاحيات الكائنات الممنوحة للأدوار
SELECT * FROM ROLE_TAB_PRIVS;

```

### 🎭 ج. جداول الأدوار (Roles Views):

```sql
-- لعرض جميع الأدوار (Roles) المخزنة والموجودة في قاعدة البيانات
SELECT * FROM DBA_ROLES;

-- لعرض الأدوار الممنوحة للمستخدمين أو الممنوحة لأدوار أخرى
SELECT * FROM DBA_ROLE_PRIVS;

-- لعرض الأدوار الممنوحة للمستخدم الحالي المتصل فقط
SELECT * FROM USER_ROLE_PRIVS;

-- لعرض الأدوار الممنوحة لأدوار أخرى (الأدوار المتداخلة / Nested Roles)
SELECT * FROM ROLE_ROLE_PRIVS;

```

## 📊 10. ملخص سريع لأهم المفاهيم والأوامر

| المفهوم / الأمر | الوصف الوظيفي والأمني |
| --- | --- |
| **`GRANT`** | أمر يتبع لـ DCL ويُستخدم لمنح الصلاحيات والأدوار للمستخدمين. |
| **`REVOKE`** | أمر يتبع لـ DCL ويُستخدم لسحب الصلاحيات والأدوار من المستخدمين. |
| **`System Privilege`** | صلاحية إدارية عامة في النظام، لا ترتبط بكائن محدد (مثل `CREATE USER`). |
| **`Object Privilege`** | صلاحية مرتبطة ومقيدة بكائن معين في الداتابيز (مثل `SELECT` على جدول). |
| **`WITH ADMIN OPTION`** | تُمنح مع صلاحيات النظام، وتسمح للمستلم بتمريرها دون أن تُسحب منه إذا سُحبت من المانح الأصلي. |
| **`WITH GRANT OPTION`** | تُمنح مع صلاحيات الكائنات، وتسبب سحباً متسلسلاً تلقائياً لكل السلسلة في حال إلغائها. |
| **`Role`** | حزمة برمجية تجمع مجموعة من الصلاحيات لتسهيل وإسراع عملية الإدارة الإدارية. |
| **`Secure Role`** | دور محمي بكلمة مرور ومخصص لحماية وتأمين الصلاحيات الشديدة الحساسية. |
| **`SET ROLE`** | الأمر المستخدم لتفعيل الـ Secure Role في الجلسة عبر كتابة الباسورد. |

> 📌 **تذكّري دائماً:** لغة التحكم بالبيانات **DCL (Data Control Language)** تشمل فقط الأمريين الجوهريين `GRANT` و `REVOKE` وهي المسؤولة مسؤولية كاملة عن إدارة جدار الحماية والصلاحيات داخل قاعدة بيانات أوراكل.
