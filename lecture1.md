## 🔐 1. المصادقة مقابل التفويض (Authentication vs. Authorization)

من الضروري جداً لأي مهندس قواعد بيانات معرفة الفرق الجوهري بين المفهومين:

* **المصادقة (Authentication):** هي عملية **التأكد من هوية** المستخدم (هل أنت فعلاً الشخص الذي تدّعيه؟) عن طريق كلمة المرور، نظام التشغيل، أو البصمة.
* **التفويض (Authorization):** هي عملية **تحديد الصلاحيات** وما يحق للمستخدم القيام به داخل قاعدة البيانات بعد نجاحه في الدخول.

---

## 💻 2. الدخول بدون كلمة مرور (OS Authentication)

عند نسيان كلمة مرور المستخدم `SYSTEM` أو `SYS`، تتيح أوراكل ميزة الدخول عبر مصادقة نظام التشغيل (Windows OS Authentication) لأصحاب الصلاحيات العالية.

### أمر الدخول عبر SQL*Plus:

```sql
CONNECT / AS SYSDBA

```

> 💡 **ملاحظة:** الرمز `/` يعني تسجيل الدخول باستخدام مستخدم نظام التشغيل الحالي، وهو ما يمنح أعلى صلاحية للتحكم بالقاعدة.

### تغيير كلمة مرور المستخدم بعد الدخول:

```sql
ALTER USER system IDENTIFIED BY new_password;

```

### 🛠️ شروط نجاح الاتصال عبر الـ OS:

1. أن يكون المستخدم الحالي على نظام التشغيل (Windows مثلاً) يمتلك صلاحيات **Administrator**.
2. أن يكون المستخدم عضواً في المجموعة الأمنية المحلية **`ORA_DBA`** (Local Users and Groups).
3. لنجاح أي اتصال عام بقاعدة البيانات، يحتاج المستخدم دائماً إلى: **(IP, SID, Port) + Authentication + Privileges**.

## 🛠️ 3. إنشاء المستخدمين وجدول التطبيق الخلفي

هناك فلسفتان مختلفتان في إدارة المستخدمين؛ لكل منهما مزاياها وعيوبها وفقاً لرأي المهندس:

### الطريقة الأولى: إنشاء مستخدم كـ Object داخل قاعدة البيانات

```sql
CREATE USER store_admin IDENTIFIED BY store;

-- منح الصلاحيات الأساسية أو الكاملة
GRANT CREATE SESSION TO store_admin;
GRANT RESOURCE TO store_admin;
-- أو إعطاؤه صلاحيات كاملة كمشرف:
GRANT DBA TO store_admin;

```

* **أمر حذف المستخدم بكل محتوياته (جداول، سيكونس... إلخ):**
```sql
DROP USER T11 CASCADE;

```



### الطريقة الثانية: نهج جدول التطبيق في الخلفية (يفضلها المبرمجون)

يتم إنشاء جدول عادي داخل قاعدة البيانات لتخزين بيانات المستخدمين وسجلاتهم:

```sql
CREATE TABLE users_tb (
  user_id   NUMBER,
  password  VARCHAR2(150),  -- يجب أن تكون مشفرة (Hashed)
  status    NUMBER,         -- 1 نشط، 0 غير نشط
  cr_date   DATE
);

```

* **آلية التحقق عند تسجيل الدخول:**
```sql
SELECT COUNT(*) FROM users_tb
WHERE user_id   = :username
  AND password  = ENCRYPT(:entered_password)
  AND status    = 1;

```


* إذا كان الناتج = **1** ←  **دخول ناجح**
* إذا كان الناتج = **0** ← ❌ **فشل الدخول**



> ⚠️ **تحذير أمني هام:** يُمنع منعاً باتاً تخزين كلمات المرور كنصوص صريحة (Plain Text) داخل الجدول، بل يجب تشفيرها حتماً باستخدام خوالزميات الـ Hash.

## 📁 4. مفهوم الـ Tablespace وعزل بيانات المستخدمين

الـ **Tablespace** هو الإطار المنطقي (Logical Framework) لتخزين البيانات الفيزيائية في أوراكل. بدلاً من حشر بيانات جميع المستخدمين في المساحة الافتراضية المسمّاة `USERS`، يتم إنشاء مساحات معزولة لكل تطبيق لتنظيم العمل وتحسين الأداء.

### أوامر إدارة الـ Tablespace والمستخدمين:

```sql
-- 1. إنشاء Tablespace جديد بمساحة 10 جيجابايت
CREATE TABLESPACE ucas_tbs
  DATAFILE 'ucas.dbf' SIZE 10G;

-- 2. إنشاء مستخدم وربطه بالـ Tablespace وتحديد كوتا (مساحة مسموحة)
CREATE USER Ahmed IDENTIFIED BY pass123
  DEFAULT TABLESPACE ucas_tbs
  QUOTA 100M ON ucas_tbs;

-- 3. تغيير كلمة مرور المستخدم
ALTER USER Ahmed IDENTIFIED BY new_password;

-- 4. قفل الحساب
ALTER USER Ahmed ACCOUNT LOCK;

-- 5. فتح الحساب
ALTER USER Ahmed ACCOUNT UNLOCK;

-- 6. حذف المستخدم نهائياً مع كافة بياناته ومحتوياته
DROP USER Ahmed CASCADE;

```

## ⚙️ 5. طرق المصادقة الثلاث (Authentication Methods) والـ Parameters

تتيح أوراكل ثلاثة خيارات رئيسية للتحقق من الهوية:

1. **Password Authentication:** الطريقة التقليدية والأكثر انتشاراً (ID وكلمة مرور داخل القاعدة). ويمكن استبدالها في التطبيقات الحديثة ببصمة الإصبع أو الوجه كبديل رقمي للتحقق الهووي.
2. **External Authentication (OS Auth):** تعتمد على مفهوم الدخول الموحد (**SSO**). إذا دخلت على الويندوز أو الدومين، تتعرف عليك أوراكل تلقائياً بدون طلب باسوورد مجدداً عبر الأمر `CONNECT /`.
3. **Global Authentication:** تشبه المصادقة الخارجية ولكنها مخصصة للأنظمة الموزعة (Distributed Databases) حيث يتم ربط مستخدمي الدومين باستخدام بروتوكول **LDAP** (مثل ربط فروع بنك فلسطين بقاعدة بيانات موحدة).

### ⚙️ أهم الباراميترات (Parameters) وطريقة تعديلها:

لعرض الباراميترات المتعلقة بنظام التشغيل من داخل الـ SQL*Plus:

```sql
SHOW PARAMETER os;

```

* **`REMOTE_OS_AUTHENT`:** المسؤول عن السماح بالمصادقة الخارجية عن بُعد عبر الشبكة. قيمته الافتراضية `FALSE`.
* **`OS_AUTHENT_PREFIX`:** البادئة التي يجب أن يبدأ بها اسم المستخدم الخارجي في أوراكل (الافتراضي هو `OPS$`).

#### طريقة التعديل الاحترافية (تتطلب إعادة تشغيل):

```sql
ALTER SYSTEM SET remote_os_authent = TRUE SCOPE=SPFILE;
ALTER SYSTEM SET os_authent_prefix = 'UCAS$' SCOPE=SPFILE;

-- إعادة تشغيل الداتابيز إجبارياً لتطبيق التعديلات
STARTUP FORCE;

-- إنشاء مستخدم بالبادئة الجديدة مأخوذ من يوزر الويندوز (whoami)
CREATE USER "UCAS$desktop-sl0qutr\click" IDENTIFIED EXTERNALLY;

```

> 🛑 **ثغرة أمنية خطيرة:** تفعيل `REMOTE_OS_AUTHENT = TRUE` خطير جداً؛ لأن الداتابيز تثق بأي جهاز بعيد. لو قام مخترق بإنشاء يوزر Admin على لابتوبه الشخصي بنفس الاسم، ستسمح له الداتابيز بالولوج فوراً! لذلك اعتبرت شركة أوراكل هذا الباراميتر **ملغياً تماماً (Deprecated/Removed)** في الإصدارات الحديثة مثل 19c و 21c لحماية الشركات.

## 👥 7. الملفات الشخصية وإدارة الموارد (User Profiles)

الـ **Profile** هو كائن يتم إنشاؤه لترجمة سياسة أمن المعلومات (**Security Policy**) الخاصة بالمؤسسة وتحويلها إلى أوامر تقنية تطبق على مستخدمي قاعدة البيانات.

### الترتيب الإداري لتطبيق الحماية:

$$\text{المؤسسة تقر البوليسي} \longrightarrow \text{الـ DBA يحولها لـ Profile} \longrightarrow \text{يتم ربط الـ Profile بالـ User}$$

ينقسم التحكم داخل الـ Profile إلى مسارين أساسيين:

### 1️⃣ مسار إدارة الموارد (Resource Control):

يتحكم في حجم استهلاك المستخدم من موارد السيرفر، مثل:

* حجم استهلاك المعالج والـ CPU.
* زمن الخمول المسموح (**Idle Time**) قبل فصل الجلسة تلقائياً.
* عمليات القراءة والكتابة على الهارد ديسك (I/O).
* المساحة المتاحة للمستخدم داخل ذاكرة الـ SGA.
* **معايير اليوزرات:** يتم التفريق بين مستخدم **Parent (Owner)** الذي يمتلك الكائنات وله معايير استهلاك مرنة، وبين **المستخدم العادي (Read/Write)** الخاضع لرقابة صارمة.

### 2️⃣ مسار أمن كلمات المرور (Password Control):

يتحكم في خصائص الأمان الحيوية للحساب:

* **Password History:** كم عدد المرات التي يُمنع فيها إعادة استخدام الباسوورد القديمة.
* **Complexity Verification:** مدى تعقيد كلمة المرور وقوتها.
* **Expiration:** متى تنتهي صلاحية الباسوورد وتجبر الداتابيز المستخدم على تغييرها.
* **Account Lock:** بعد كم محاولة دخول خاطئة يتم قفل الحساب تلقائياً.

## 📊 8. مستويات قاموس البيانات (Data Dictionary Views)

ينقسم قاموس البيانات في أوراكل إلى ثلاثة مستويات تصاعدية من حيث الصلاحيات:

1. **`DBA_` (مستوى المشرف الكامل):** يعرض كل الكائنات (Objects) الموجودة في قاعدة البيانات بالكامل بغض النظر عن مالكها. لا يمكن الاستعلام منه إلا بصلاحية DBA، وإلا سيظهر خطأ تفيد بأن الجدول غير موجود `ORA-00942`.
```sql
SELECT COUNT(*) FROM dba_objects;

```


2. **`ALL_` (مستوى الصلاحية المتاحة):** يعرض كل الكائنات التي يمتلك المستخدم الحالي صلاحية الوصول إليها، حتى لو لم يكن هو المالك الأصلي لها.
```sql
SELECT COUNT(*) FROM all_objects;

```


3. **`USER_` (مستوى الملكية الشخصية):** يعرض فقط الكائنات التي يملكها المستخدم الحالي نفسه (`Owner`).
```sql
SELECT COUNT(*) FROM user_objects;

```



### أوامر التحكم بصلاحيات الـ DBA ومراقبة البروفايل:

```sql
-- سحب صلاحية الـ dba من مستخدم معين (سيمنعه من قراءة جداول dba_)
REVOKE dba FROM "UCAS$DESKTOP-SL0QUTR\CLICK";

-- إعادة منح الصلاحية مجدداً
GRANT dba TO "UCAS$DESKTOP-SL0QUTR\CLICK";

-- معرفة اسم البروفايل المربوط بمستخدم معين
SELECT profile FROM dba_users WHERE username = UPPER('UCAS$desktop-sl0qutr\click');

-- فحص الشروط والحدود الحالية داخل البروفايل الافتراضي (DEFAULT)
SELECT resource_name, limit FROM dba_profiles WHERE profile = 'DEFAULT';

```

## 🔑 9. سياسات أمن كلمات المرور وحظر الحسابات (تطبيق عملي)

### تجربة تعديل محاولات الدخول الفاشلة إلى 5 محاولات:

```sql
-- 1. إنشاء بروفايل مخصص وتحديد سقف المحاولات الخاطئة بـ 5 محاولات
CREATE PROFILE ucas_profile LIMIT FAILED_LOGIN_ATTEMPTS 5;

-- 2. ربط البروفايل الجديد بالمستخدم المطلوب
ALTER USER "UCAS$DESKTOP-SL0QUTR\CLICK" PROFILE ucas_profile;
ALTER USER WESAM PROFILE ucas_profile;

```

> 🧪 **نتيجة التجربة العمليّة:** عند محاولة تسجيل الدخول 5 مرات متتالية بكلمة مرور خاطئة، يقوم النظام بإغلاق الحساب فوراً ويظهر الخطأ الشهير: **`ORA-28000: the account is locked`**.

* **أمر فك قفل الحساب وإعادة تفعيله:**
```sql
ALTER USER WESAM ACCOUNT UNLOCK;

```



### 🗓️ انتهاء صلاحية كلمة المرور (Password Expiration):

عند ضبط فترة الصلاحية (مثلاً `PASSWORD_LIFE_TIME 180` والتي تعني 180 يوماً، أي حوالي 6 أشهر)، ستقوم قاعدة البيانات تلقائياً بمطالبة المستخدم بتغيير الباسوورد فور انتهائها، وتبدأ في إرسال رسائل تحذيرية قبل أسبوع تقريباً من موعد الانتهاء.

## 🛡️ 10. التحذيرات الأمنية المتقدمة واستراتيجيات الحماية

### 🚨 خطر هجمات حجب الخدمة (DoS) الناتجة عن قفل الحسابات:

المبرمج التقليدي يظن أن قفل الحساب بعد 5 محاولات فاشلة هو حماية مطلقة، ولكن المهندس ينبه إلى **ثغرة خطيرة**: لو عرف المهاجم (Hacker) اسم مستخدم موظف رئيسي أو مدير النظام، فسيقوم عمداً بإدخال باسوورد خاطئة 5 مرات في أقل من دقيقة، مما يتسبب في قفل حساب الموظف الحقيقي وتعطيل أعمال الشركة دون اختراق أي شيء! هذا يسمى هجوم حجب الخدمة (DoS).

#### 🛡️ الحلول البديلة والاحترافية:

1. **حظر الـ IP وليس المستخدم:** مراقبة الأجهزة التي تصدر منها المحاولات الفاشلة وحظر الـ IP الخاص بها (عبر الـ Firewall أو قاعدة البيانات) مع ترك حساب الموظف يعمل بشكل طبيعي.
2. **نظام التنبيهات الفورية:** ربط النظام بآلية ترسل رسائل نصية (SMS) أو تنبيهات فورية لفريق الأمن السيبراني (SOC/Administrator) عند رصد محاولات تخمين باسوورد نشطة قبل أن يُغلق الحساب.

---

### 🔑 ميزة التحقق من تعقيد كلمة المرور الجاهزة:

توفر أوراكل دالة مدمجة داخل النظام جاهزة ومبرمجة تُدعى **`VERIFY_FUNCTION_11G`**. بمجرد ربطها بالبروفايل، تتولى قاعدة البيانات فحص شروط التعقيد تلقائياً (مثل: احتواء الباسوورد على رموز، أرقام، أحرف كبيرة، ألا تكون مقلوبة أو مطابقة لاسم المستخدم) مما يغنيك عن كتابة آلاف الأسطر البرمجية للتحقق يدويّاً.

---

### 🚪 تأمين الحسابات الافتراضية والأبواب الخلفية (Back Doors):

عند تثبيت أوراكل، تنزل تلقائياً حسابات تجريبية وافتراضية مشهورة عالمياً (مثل يوزر `SCOTT` وباسوورده الافتراضي `tiger`). يحذر المهندس بشدة من تركها مفتوحة لأنها تمثل **باباً خلفياً سهلاً** للمخترقين لتدمير النظام.

* **الحل الفوري:** قفلها تماماً أو تغيير باسوورداتها الافتراضية فوراً.

---

### 🛑 حظر حذف المستخدم المتصل:

قاعدة صارمة في أوراكل: **لا يمكن نهائياً** عمل `DROP USER` لأي مستخدم طالما يمتلك جلسة نشطة حالياً ومتصل بقاعدة البيانات (`Currently Connected`). يجب إنهاء جلسته أولاً قبل الحذف.

---

### 🔒 تشفير الباسوورد أثناء النقل والتخزين:

تنتقل كلمات المرور من جهاز العميل (Desktop/Web Application) إلى قاعدة البيانات مشفرة بالكامل، وتُخزن داخل قاعدة البيانات مشفرة باستخدام خوارزمية التشفير العالمية المتطورة **AES**. الباسوورد لا تمر في الشبكة كنص واضح ولا تُخزن صريحة، لقطع الطريق تماماً على أي محاولات للتجسس وسرقة الحزم من الشبكة (**Sniffing**).
